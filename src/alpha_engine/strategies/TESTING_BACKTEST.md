# TESTING_BACKTEST: Microstructure Modeling & Deterministic Performance Metrics

This document defines the high-fidelity microstructure modeling parameters, performance metric equations, and debugging procedures for the **BullDog Alpha** offline backtesting engine.

---

## 1. High-Fidelity Microstructure Modeling

To bridge the gap between backtesting and real-world execution, the backtest simulator enforces slippage and commission calculations on every transaction fill.

### 1.1 Fixed Fee Model (Brokerage Commission)
Every executed fill incurs a proportional rate commission plus a flat ticket fee:

$$\text{Commission} = \text{Order Qty} \times \text{Execution Price} \times \text{Commission Rate} + \text{Flat Fee}$$

* **Default Commission Rate**: $0.0001$ ($0.01\%$ or $1$ basis point).
* **Default Flat Fee**: $\$1.00$ per transaction fill.

### 1.2 Linear Volume Slippage Model
Slippage models market impact cost dynamically depending on the order size relative to the bar's traded volume:

$$\text{Execution Price} = \text{Bar Price} \times \left(1 \pm \gamma \cdot \frac{\text{Order Qty}}{\text{Bar Volume}}\right)$$

* For `BUY` orders, the price is shifted upward (plus sign):
  $$\text{Exec Price} = \text{Bar Price} \times \left(1 + \gamma \cdot \frac{\text{Order Qty}}{\text{Bar Volume}}\right)$$
* For `SELL` orders, the price is shifted downward (minus sign):
  $$\text{Exec Price} = \text{Bar Price} \times \left(1 - \gamma \cdot \frac{\text{Order Qty}}{\text{Bar Volume}}\right)$$

Where:
* $\gamma$ (Gamma) represents the impact scaling coefficient. Default is $0.1$.
* If $\text{Bar Volume} = 0$, slippage is treated as zero to prevent division by zero errors.

## 2. Quantitative Performance Metrics

### 2.1 Net Asset Value (NAV)
At each event-time step $t$, the portfolio Net Asset Value (NAV) is computed as:

$$\text{NAV}_t = \text{Cash}_t + \sum_{s} \text{Qty}_{s,t} \times \text{Price}_{s,t}$$

### 2.2 Sharpe Ratio (Annualized)
The Sharpe Ratio measures risk-adjusted returns. For minutely backtests, returns are calculated minute-by-minute:

$$R_t = \frac{\text{NAV}_t - \text{NAV}_{t-1}}{\text{NAV}_{t-1}}$$

Let $\mu$ be the sample mean and $\sigma$ be the sample standard deviation of $R_t$:

$$\text{Sharpe Ratio} = \frac{\mu}{\sigma} \times \sqrt{252 \times 390}$$

* The scaling factor $\sqrt{252 \times 390}$ annualizes the minutely volatility based on 252 trading days per year and 390 trading minutes per day (9:30 AM to 4:00 PM EST).
* If $\sigma = 0$, the Sharpe Ratio defaults to $0.0$.

### 2.3 Sortino Ratio
The Sortino Ratio evaluates risk-adjusted returns relative to downside deviation:

$$\sigma_d = \sqrt{\frac{1}{M-1} \sum_{t=1}^M (\min(R_t, 0))^2}$$

$$\text{Sortino Ratio} = \frac{\mu}{\sigma_d} \times \sqrt{252 \times 390}$$

* If $\sigma_d = 0$, the Sortino Ratio defaults to $0.0$.

### 2.4 Maximum Drawdown
Maximum Drawdown represents the largest peak-to-trough drop in NAV:

$$\text{Max Drawdown} = \max_{t} \left( \frac{\max_{\tau \le t} \text{NAV}_\tau - \text{NAV}_t}{\max_{\tau \le t} \text{NAV}_\tau} \right)$$

## 3. Determinism Debugging Checklist

If successive backtests on identical Parquet datasets yield performance differences $> 10^{-9}$ in float64 memory representation, use the following checklist to isolate state leaks:

1. **Garbage Collection**:
   Ensure `gc.collect()` is run explicitly before launching the backtest session.
2. **Strategy Local State**:
   Verify that all strategy member variables, arrays, indicators, and rolling window buffers (such as deques or lists) are initialized in the `__init__` constructor and not leaked or shared between backtest iterations.
3. **Random Seeds**:
   If the strategy employs non-deterministic algorithms (e.g. reinforcement learning agents, stochastic optimization), explicitly pin standard library seeds:
   ```python
   import random
   import numpy as np
   random.seed(42)
   np.random.seed(42)
   ```
4. **Polars Sorting**:
   Verify that `pl.scan_parquet().sort("timestamp")` sorts the input chronologically. Multi-symbol backtests must be grouped chronologically to avoid timestamp interleaving mismatches.
5. **Timezone Uniformity**:
   All timestamps generated by `cmd/data_recorder` are stored in UTC milliseconds since Epoch. Strategy time calculations must strictly operate in UTC to prevent daylight savings shifts or local server timezone variance.
